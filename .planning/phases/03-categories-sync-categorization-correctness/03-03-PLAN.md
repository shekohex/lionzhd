<task type="auto">
  <name>Task 1: Implement SyncCategories action with correctness rules</name>
  <files>app/Actions/SyncCategories.php</files>
  <action>
Create `App\Actions\SyncCategories` using the existing AsAction pattern.

Behavior (must match Phase 03 context decisions):

Fetching:
- Fetch BOTH VOD and Series categories from Xtream via the new Saloon requests.
- Fetch each source independently with isolated try/catch so one side failing does not prevent the other side applying.
- Treat "0 categories" as a special case that requires explicit confirmation before applying destructive changes.
- The action must accept inputs to allow applying an empty fetch when explicitly confirmed (e.g. `forceEmptyVod` / `forceEmptySeries`).
- Normalize each category row to:
  - `provider_id` as string (skip rows with empty/missing IDs; record warning)
  - `name` as string (default empty allowed but warn)

Identity + updates:
- Category identity is `provider_id`.
- If same provider_id returns with a renamed title, update stored `name` to provider value.
- If duplicates exist by name with different IDs, keep both (no merge).
- Same provider_id across VOD and Series is a single shared category: set `in_vod` and/or `in_series` flags accordingly.
- Scope-flag safety in partial runs:
  - Update `in_vod` only when VOD fetch succeeded and is allowed to apply.
  - Update `in_series` only when Series fetch succeeded and is allowed to apply.
  - If a source failed or was blocked by empty confirmation, preserve existing flag for that source.

Missing categories cleanup:
- Compute provider-id sets per source (`vodIds`, `seriesIds`) only for sources that successfully fetched.
- Do NOT perform hard-delete cleanup when a source failed OR a source returned 0 categories without explicit force for that source.
- When both sources successfully fetched (and any empty-source was explicitly forced), then:
  - Compute union of provider IDs from current sync results (vod + series).
  - Hard remove categories (non-system) whose provider_id is NOT in the union.

Uncategorized handling:
- Ensure system categories exist (by provider_id constants) for VOD and Series.
- Apply per-source content moves ONLY when that source successfully fetched and is allowed to apply (not blocked by empty-source confirmation).
- For VOD content (`vod_streams`): any row whose `category_id` is null/empty OR not present in the *current valid VOD category set* must be auto-assigned to VOD Uncategorized.
- For Series content (`series`): same, but Series Uncategorized.
- When moving a row to Uncategorized, set `previous_category_id` to the prior `category_id` (if present) so remap can occur later.

Auto-remap on reappearance:
- Apply per-source remap ONLY when that source successfully fetched and is allowed to apply.
- If a category reappears (provider_id now present in the valid set), then for rows currently in that type's Uncategorized where `previous_category_id` matches that provider_id: set `category_id` back to provider_id and clear `previous_category_id`.

Partial success semantics (context decision):
- If one source succeeds and the other fails, keep successful-side changes and record a partial outcome with issues for the failed side.
- In partial success runs, do not run global destructive cleanup that could remove categories based on incomplete data.

Run recording:
- Create a `CategorySyncRun` row per execution; store:
  - status transitions (queued/running/success/success_with_warnings/failed)
  - summary counts (created/updated/removed, moved_to_uncategorized_vod/series, remapped_from_uncategorized_vod/series)
  - top_issues array (invalid rows skipped, source fetch failures, empty-source confirmation required)

Return value: return the `CategorySyncRun` id (or the model) for UI/history linkage.
  </action>

<task type="auto">
  <name>Task 2: Add SyncCategories queued job with single-run lock and requeue behavior</name>
  <files>app/Jobs/SyncCategories.php</files>
  <action>
Create a queued job that runs SyncCategories.

Concurrency rule (from context): if a sync is already running, queue the next run.
Implement by using a cache lock inside `handle()`:
- Try acquiring a lock key (e.g. `sync:categories`) with a TTL.
- If lock cannot be acquired: `release(<short delay>)` so the job is retried later (do NOT discard).
- If acquired: run `SyncCategories::run()` and release lock in finally.

Do NOT use `ShouldBeUnique` in a way that prevents multiple dispatches from being queued.

Support empty-source confirmation:
- The job must accept inputs (e.g. `forceEmptyVod`, `forceEmptySeries`) so the controller can dispatch a forced run after the user confirms.
- The job passes those inputs to `SyncCategories::run(...)`.

  <done>Multiple dispatches are allowed; execution is serialized by lock+release, not by unique-job dropping.</done>
